# 手机端 SPA → 多页面改造计划

## 现状

- 18 个页面全部塞在 `index-mobile.html` 一个文件里，通过 `showPage()` 切换
- 共享 JS 状态：`mobile` 对象（currentUser, photos, categories 等全部在内存）
- 共享 CSS：`mobile.css` 单一文件
- 共享依赖：Supabase SDK、Leaflet、JSZip 等 CDN 引入
- 版本管理：通过 `?v=N` 缓存破坏

## 分步实施计划

### 第1步：抽取公共层（关键基础，约2小时）

**目标**：创建 `shared/` 目录，把公共代码从 `mobile-app.js` 抽出来

1. **`shared/supabase-auth.js`**
   - `initSupabase()` — 初始化 Supabase 客户端
   - `checkLogin()` — 检查 localStorage 是否有 session，无则跳转 login.html
   - `handleLogout()` — 清除 session 跳回 login.html

2. **`shared/core.js`**
   - 公共状态管理：从 `mobile` 对象迁移到 `window.AppState`
     ```
     window.AppState = {
       currentUser, photos, categories, photoCategories,
       currentPage, showFavoritesOnly, ...
     }
     ```
   - 公共方法：`loadCategories()`, `loadAllPhotoCategories()`, `getPhotoUrl()`, `escapeHtml()`, `highlightKeywords()`

3. **`shared/components.css`**
   - `.top-bar` 顶部导航栏（含 ← 返回按钮逻辑）
   - `.bottom-nav` 底部导航栏
   - `.modal-overlay`, `.modal-card` 弹窗
   - `.toast` 提示
   - `.empty-state`, `.loading`

### 第2步：拆分页面（逐个迁移，每个约30分钟）

按优先级（使用频率高的先做）：

**第一批 — 核心页面（4个）**

## 困难与风险分析

### 🔴 高风险

**1. 状态共享丢失**
- 现在所有数据在 `mobile` 对象内存中，跨页面后丢失
- 必须用 `localStorage` 或每次重新从 Supabase 拉取
- 影响：用户在地图页筛选的分类跳转到照片页后筛选丢失
- 方案：只在 `localStorage` 存 `currentUser`、`session` 等核心状态；数据每次从 Supabase 拉取（反正有缓存）

**2. Supabase Session 跨页面**
- `supabase.auth.getSession()` 默认存在 localStorage，跨页面能读取
- 但需要每个页面加载后调 `getSession()` 恢复登录态
- 风险：首次加载慢，可能闪现登录页
- 方案：`checkLogin()` 在 `<head>` 中同步执行，未登录直接 `location.replace('/login.html')`

**3. CDN 依赖加载顺序**
- Supabase SDK、Leaflet 等 CDN 脚本每个页面都要引入
- 如果某个页面引漏了，功能直接挂
- 方案：创建模板文件 `_template.html` 包含所有必需的 `<script>` 和 `<link>`

### 🟡 中风险

**4. 代码重复**
- 每个页面需要独立的 CRUD 逻辑（如心情日记的增删改查）
- 方案：页面级逻辑留在页面内，公共的放 `shared/`；不追求零重复

**5. 文件数量暴增**
- 18 个 HTML 文件 + 多个 shared JS/CSS，维护成本上升
- 方案：用一个 `build.js` 或 shell 脚本统一改版本号

**6. CSS 碎片化**
- 现在 `mobile.css` 包含所有页面样式，拆分后每个页面加载会用不到CSS
- 方案：`shared/components.css` + 每个页面一个 `<style>` 块（或保持 `mobile.css` 只拆公共部分）

### 🟢 低风险

**7. 版本号管理**
- 现在改一个 `?v=N` 就行，之后要改 18 个文件
- 方案：`shared/core.js` 定义 `APP_VERSION`，各页面读取它追加到 URL 参数

**8. Service Worker 缓存策略**
- 现有 `sw.js` 可能缓存旧的 SPA 资源导致多页面加载异常
- 方案：更新 `sw.js` 排除 `.html` 文件不缓存，或用 Network First 策略

**9. 桌面版不受影响**
- `index.html`(桌面版) 保持 SPA 不变，本次只改手机端

## 决策点（需要你选）

1. **跳转方式**：`location.href`（带页面刷新，简单可靠） vs `location.replace`（不能后退，防止回退到中间态）
2. **详情页传参**：URL 参数 `?photoId=xxx`  vs  `localStorage` 临时存储
3. **是否需要保留 SPA 版本**：保留 `index-mobile-spa.html` 作为兜底，还是直接删掉
4. **底部导航**：4 个 `<a>` 链接始终显示，还是在某些页隐藏（如登录、详情）
