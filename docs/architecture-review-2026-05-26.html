Problem

Two parallel code paths for the same database operations. mobile-app.js imports photo-service.js but reimplements loadPhotos, toggleFavorite, loadAllPhotoCategories inline — imports are dead code.

Solution

Make mobile-app.js delegate all data access through src/core/ services. Delete the inline reimplementations.

Wins

  • ~200 lines deleted from mobile-app.js
  • single seam for Supabase calls
  • dead imports eliminated

2. Extract auth & theme from the god object

Strong in-process

Problem

Auth (login/logout/password change, ~150 lines), theme (dark/warm/blue, ~100 lines), and birthday easter egg (~120 lines) are tangled into the 2058-line god object. These are self-contained concerns with no cross-dependencies.

Solution

Extract auth-module.js, theme-module.js, birthday-module.js. Each takes a reference to mobile (not the full object). Loadable via the existing _ensureModule mechanism.

Wins

  • mobile-app.js drops ~370 lines
  • locality: auth bugs isolated to auth-module
  • each module testable independently
  • consistent with existing lazy-load pattern

3. Deepen photo-service and storage-service

Worth exploring in-process

Problem

photo-service exports 9 functions but only 1 (loadPhotos) has meaningful logic; the rest are single-line supabase delegation. storage-service's uploadFile is a pass-through wrapper. Both fail the deletion test: if removed, callers just call supabase directly with no complexity increase.

Solution

Merge photo-service into a single deep module with a narrower interface. Inline storage-service — its 2 exports provide zero leverage. Add error normalization, retry logic, and cache invalidation behind the seam.

Wins

  • interface shrinks 9→4 entry points
  • leverage: error handling, retry, cache at the seam
  • storage-service: deletion test passed — remove it
  • locality: all Supabase photo calls behind one seam

Solution

Wins

5. Shared modal factory

Worth exploring in-process
modules/photos-module.js · modules/diary-module.js · modules/records-module.js · modules/albums-module.js · modules/extras-module.js

Before: 7+ modules each create modals by hand

createElement('div')
className = 'modal-overlay'
innerHTML = `<div class="modal-card">...`
appendChild(modal)
zIndex = '1000'

↑ repeated verbatim in every module

After: one modal factory at the seam

modalFactory.show({
title, body, actions
})

↑ single interface, ~40 lines saved per modal

Problem

Every module reimplements modal creation: createElement, set className, build innerHTML template, append to body, manage z-index. ~15 duplicate modal implementations across 7 modules.

Solution

Add modalFactory to mobile-app.js with a single interface: show({title, body, actions}), close(). Modules call mobile.showModal(...) instead of building DOM manually.

Wins

  • leverage: one interface, 15+ call sites
  • locality: modal z-index/layout at one seam
  • fix modal bug once, every module benefits
  • ~200 lines deleted across modules

6. Desktop views: collapse window.* + export duality

Speculative in-process
src/desktop/views/*.js (12 files) · app.js (2734 lines) · src/desktop/main.js (189 lines)

Before: Dual registration per view

window.getPhotoUrl = ...
window.openDetail = ...
window.toggleFavorite = ...

152× window.* assignments

export function getPhotoUrl
export function openDetail
export function toggleFavorite

40× export statements

After: Single registration pattern

export manager = {
getPhotoUrl,
openDetail,
toggleFavorite
}

one export per view

window.* populated once by main.js

Problem

12 desktop views each maintain 15-26 window.* assignments AND matching export blocks — identical name lists maintained in parallel. Changing one function requires updating both lists. 152 window.* + 40 export = 192 registration points.

Solution

Each view exports a single manager object. main.js iterates over the manager and auto-populates window.* from exports. No manual dual maintenance.

Wins

  • leverage: main.js registers all window.* centrally
  • add a function → add it once, auto-registered
  • deletion test: window.* block = export block = waste
